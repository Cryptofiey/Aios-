# AIOS UI Framework Standards & Architecture

Эта документация фиксирует визуальные и архитектурные правила, которых должны придерживаться агенты-разработчики при создании новых UI элементов для Web Agent OS (T.A.E. Combine).

## 1. Парадигма Пространства (Space Conservation)

Критическое правило: **Тотальная экономия пространства.**
Устройство должно ощущаться как терминал управления или плотная приборная панель космического корабля/кибер-деки. 

* **Минимум бесполезного пространства (Padding/Margin):** 
  Все внешние рамки должны быть под самые края дисплея (например `p-1`, а не `p-4` или `p-6`). Внутренние боксы располагаются под края рамок.
* **Межклеточное пространство:** Оставлять ровно столько пространства (gap: `1`, `0.5`, или `2` максимум), чтобы глазу хватало увидеть логическое разделение между частями интерфейса. Никаких гигантских отступов ради "воздуха".
* **Размеры шрифтов:** Использовать компактную типографику (`text-[10px]`, `text-xs`).

## 2. Структура Окон (OSWindow & ShellMatrix)

Все новые UI-модули и приложения-песочницы должны оборачиваться в стандартизированные контейнеры. Приложение не существует "просто в div", оно живет внутри эмулированного оконного менеджера.

### 2.1 OSWindow
Используется для стандартных системных панелей, таких как панель инструментов, терминалы сборок, отображения дерева файлов:
* **Атомарные блоки:** Внутри OSWindow UI часто разбивается на сетку мини-модулей (Mod_1, Mod_2 и тд).
* **Стилизация:** Темный фон (`bg-black`, `bg-slate-900/50`), тонкие рамки (`border-slate-800`), использование `backdrop-blur`.

### 2.2 ShellMatrix
Используется для рендеринга слоистых, комплексных приложений (когда приложение генерируется из NPM компонентов, собирается через Git, и компилируется в UI-слой "APP").
* **Left Panel (SMART VIEW):** Вертикальная панельная зона (w-6 / w-8), содержащая цветовые и символьные индикаторы состояния агента (`thinking`, `executing`, `success`). 
* **Верхние Слои сборки:** Вкладки (`APP`, `PKG`, `GIT`, `NPM`, `BASH`). Олицетворяют концепцию атомарной блочности.
* **Original Shell:** Главный прямоугольник вывода кода, окруженный "Умной Обводкой", которая пульсирует в зависимости от состояния агента. 

## 3. Навигация (ViewportManager)

Переключение рабочих столов (Workspaces) осуществляется через `ViewportManager` исключительно системно или свайпами (swipe) на сенсорных экранах или мышкой (grab).
* **Свайп логика:** Поддерживается Framer Motion `drag="x"`. Пороги свайпа настроены на минимальное сопротивление для отзывчивости (`dragElastic={1}`, быстрые `transition={{ duration: 0.15 }}`).
* **Пагинация/Док:** Нижний док максимально сужен (`h-8`), кнопки-индикаторы сжаты в минимальные прямоугольники (`h-1.5 w-8`/`w-2.5`). Никаких массивных навигационных кнопок (стрелок) по бокам дока.

## 4. Концепция Атомарной Блочности и Уровней сборки

Если агент создает приложение для пользователя:
1. Запрашиваются NPM-зависимости, они собираются в низкоуровневом слое (`NPM Shell`).
2. Код пишется и коммитится в файловую систему (`Git Shell`).
3. Виртуальная среда формирует связи компонентов (`PKG Shell`).
4. Наконец, готовое интерактивное приложение отдается в клиент (`APP Shell`).
Визуально это должно отображаться пользователю внутри компонента `ShellMatrix`.

## 5. Дополнительные UI Принципы

* **Отзывчивость:** Не использовать тяжеловесных CSS-анимаций, где их не должно быть. Использовать `will-transform` на свайпах. 
* **Анимации ввода:** Мгновенный фидбэк `hover:`, использование `active:scale-95` с минимальной задержкой. Окна появляются плавно, но быстро (`duration: 0.2` - `0.3`).
* **Толщина рамок:** Всегда `border-width: 1px`, скругления `rounded` или `rounded-md`, крайне редко `rounded-xl`.

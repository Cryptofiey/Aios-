# Web Agent OS: Request for Comments (RFC) & Architecture Analysis

Этот документ служит черновиком для анализа реализуемости, выявления рисков и проработки архитектуры Web Agent OS до того, как мы развернем автономного агента. Мы используем этот файл для проверки наших идей на "прочность".

## 1. Концепция и Парадигма
- **Ядро:** Браузер (V8), никаких классических серверов. 
- **Память (Диск):** GitHub репозитории.
- **Дисплей:** Эмулятор AI Studio (и iframe внутри него).
- **Разум:** Sema Soul Combine (для управления контекстом) + ИИ Агент-Супервизор.
- **Связь:** Агент (чат) = Терминал. Эмулятор (превью) = Рабочий стол / Дашборд.

### Философия Эфемерности и Кросс-платформенности (Termux-paradigm)
- **Низкий уровень:** GitHub и npm используются как базовые инструменты ядра просто потому, что они нативны для ИИ-агентов. Но это лишь архитектурный фундамент.
- **Динамическая изоляция без полной эмуляции:** Если пользователь запрашивает приложение, зависящее от другой ОС (например, Android), ОС не пытается запустить "тяжелую" виртуальную машину целиком. Запускается только минимальный набор необходимых пакетов и зависимостей (эмуляция узкой среды исполенния, а не железа/ОС), достаточный для изоляции и выполнения конкретной задачи.
- **Агностика к стеку:** Система не привязывает себя к технологиям жестко. Любой софт, среда или инструмент скачивается под конкретную задачу и уничтожается/кэшируется, когда более не нужен.

### Архитектура "Матрёшки" (Атомарная сборка на лету)
- **Многоуровневые контейнеры:** Пользовательский опыт строится по принципу матрёшки. Мелкие элементы упаковываются в пакеты (npm), пакеты — в модули/библиотеки (боксы), библиотеки — в приложения (git-репозитории). Сама Web Agent OS — это самая большая внешняя коробка.
- **Отказ от низкоуровневых систем:** Из концепции исключаются все классические низкоуровневые связи ОС (файловые дескрипторы, драйверы устройств). Всю "работу вертушек" (хранение, управление версиями, адресация) берет на себя Git и инфраструктура пакетов.
- **Сборка на лету:** Приложения не "устанавливаются", они собираются атомарно в реальном времени из этих готовых стандартизированных блоков, распределяя элементы по нужным уровням матрёшки для конкретной задачи.

## 2. Анализ Узких Мест (Bottlenecks) и Рисков

### Риск 1: GitHub как Файловая Система (CORS и Rate Limits)
Браузер не может напрямую делать `git clone` по SSH или классическому HTTPS из-за CORS ограничений браузера. Классические GraphQL / REST API от GitHub жестко лимитированы (5000 запросов в час).
* **Возможное решение (Тестирование обязательно):** Использование `isomorphic-git` в связке с CORS-прокси (например, `cors.isomorphic-git.org`) и хранилищем `LightningFS` (поверх IndexedDB). Нам нужно протестировать, будет ли это стабильно работать с приватными токенами в эмуляторе.

### Риск 2: Эмулятор AI Studio и Service Workers
Эмулятор работает в изолированном iframe (sandbox). Service Workers, необходимые для перехвата сетевых запросов (чтобы собрать npm-пакеты "на лету" и реализовать философию Termux), могут быть заблокированы политиками безопасности платформы AI Studio (sandbox флаги `allow-same-origin`, контекст Service Worker).
* **Возможное решение (Тестирование обязательно):** Если Service Workers заблокированы, нам придется использовать фоллбэк: `import maps` + CDN (например, esm.sh) для загрузки пакетов напрямую в среду исполнения (в динамические `<iframe>` внутри эмулятора).

### Риск 3: Динамическое выполнение кода (Sandboxing)
Система должна уметь собирать и запускать клиентский код по запросу пользователя. Выполнение неизвестного кода рядом с кодом ядра ОС (где лежат наши API ключи `NVIDIA_API_KEY`, `GITHUB_TOKEN`) критически опасно.
* **Возможное решение:** Двухуровневая изоляция. Ядро ОС (наш текущий код) имеет доступ к ключам. Когда ядро собирает пользовательское приложение, оно запускает его в строгом `<iframe>` с `sandbox="allow-scripts"`, передавая данные только через `postMessage`.

### Риск 4: "Все есть кнопка" и "Все есть цикл" в React
Генерация интерфейса ИИ агентом на лету. Агенты должны генерировать не просто текст, а React-компоненты или JSON-декларации, которые наше ядро ОС (эмулятор) рендерит как кнопки-микросервисы.
* **Возможное решение:** Супервизор и суб-агенты возвращают строго типизированный ответ (JSON Schema), который `App.tsx` проглатывает и отрисовывает как карточку задачи на дашборде.
